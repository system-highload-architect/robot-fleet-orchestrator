# ADR-002: Event-Driven архитектура

## Контекст
Система должна быть слабосвязанной, масштабируемой и устойчивой к сбоям. Модули должны взаимодействовать асинхронно, чтобы избежать блокировок и упростить распил.

## Рассмотренные варианты
- **Прямой вызов API** – синхронный, создаёт жёсткую связанность, сложно распределять.
- **Event-Driven** – асинхронный, слабая связанность, простота интеграции.

## Решение
Внедрить шину событий (`EventBus`) с абстрактным интерфейсом. В монолите использовать in-memory реализацию, при распиле – заменить на Kafka. Все модули общаются только через события.

## Последствия
- **Плюсы**: слабая связанность, гибкость, возможность замены шины без изменения кода модулей.
- **Минусы**: необходимость обработки событий, сложность отладки асинхронных цепочек, нужно обеспечить идемпотентность.

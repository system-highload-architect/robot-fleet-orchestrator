# Архитектурный документ: Robot Fleet Orchestrator

## 1. Контекст и бизнес-цели

**Заказчик**: обезличен.  
**Продукт**: платформа для управления парком роботов (роботы-доставщики, дроны, погрузчики) в складских и городских локациях.  
**Бизнес-задачи**:
- Снижение затрат на логистику за счёт автоматизации.
- Повышение скорости обработки заказов.
- Обеспечение отказоустойчивости (роботы могут терять связь, разряжаться, сталкиваться с препятствиями).
- **Интеллектуализация**: использование AI для распознавания объектов, голосовых команд и видеоаналитики, что расширяет функционал роботов (навигация, инспекция, взаимодействие с персоналом).

**Ключевые нефункциональные требования**:
- **Доступность**: 99.9% (роботы не должны простаивать).
- **Латентность**: p95 < 100 мс для команд управления.
- **Масштабируемость**: поддержка от 10 до 1000+ роботов.
- **Безопасность**: аутентификация роботов, шифрование команд, защита AI-каналов.

---

## 2. Архитектурный стиль

Выбран **модульный монолит с чёткими границами (DDD)**, который в будущем будет эволюционировать в микросервисы.

**Причины**:
- **Скорость выкатки MVP** – можно запустить за 2 месяца.
- **Минимизация рисков** – не нужно проектировать сложные распределённые транзакции на старте.
- **Гибкость** – модули изолированы, их можно выносить в отдельные сервисы без переписывания кода.

**План эволюции**:
1. Модульный монолит с in-memory шиной событий.
2. Замена шины на Kafka для подготовки к распилу.
3. Постепенный вынос модулей в микросервисы (Robot Manager, Task Dispatcher, Location Service, Analytics, **AI Processor**).

---

## 3. Ключевые сущности

- **Robot** – идентификатор, тип (доставщик/дрон/погрузчик), статус (idle/busy/offline), координаты, заряд батареи, грузоподъёмность, радиус действия, наличие камер/микрофонов.
- **Task** – идентификатор, тип (доставка/перемещение/зарядка/инспекция), приоритет, статус (pending/assigned/completed/failed), связанная локация, срок выполнения, прикреплённые файлы (изображения, видео, аудио).
- **Location** – идентификатор, тип (склад/город/зона дронов), координаты, ограничения (макс. скорость, разрешённые типы роботов).
- **Fleet** – группа роботов, работающих в одной локации; может иметь локального координатора (для роевого режима).
- **AI Module** – отдельный модуль для обработки данных с сенсоров (зрение, слух). Может работать как в централизованном режиме (на бэкенде), так и на периферии (в самом роботе) с передачей результатов в оркестратор.
- **Event** – базовое событие: RobotStatusChanged, TaskAssigned, TaskCompleted, LocationUpdated, NetworkLost, **AIObjectDetected**, **AISpeechRecognized**, **AIVideoAnalytics**.

---

## 4. Event-Driven коммуникация

Все модули общаются через **абстрактную шину событий** (`EventBus` interface).  
- **В монолите**: используется in-memory реализация (быстро, без сети).  
- **При распиле**: заменяется на Kafka (или NATS) для межсервисной коммуникации.

**События (включая AI)**:
- `robot.status.changed`
- `task.assigned`
- `task.completed`
- `location.updated`
- `network.lost`
- `ai.object.detected` (например, робот распознал препятствие или человека)
- `ai.speech.command` (голосовая команда от персонала)
- `ai.video.analytics` (анализ видеопотока: подсчёт объектов, инспекция качества)

Модули **не вызывают** друг друга напрямую – только через события.

## 5. Модули (кандидаты в микросервисы)

| Модуль | Ответственность |
|--------|-----------------|
| **Robot Manager** | Хранит состояние всех роботов, обновляет статусы, публикует события изменения состояния. |
| **Task Dispatcher** | Распределяет задачи по роботам на основе приоритетов, типов роботов и текущих нагрузок. Приоритет может повышаться при обнаружении критических событий (например, AI-детекция опасности). |
| **Location Service** | Отвечает за маршруты, ограничения локаций, предоставляет данные для диспетчера. |
| **Analytics** | Собирает метрики, строит отчёты, выявляет аномалии. |
| **AI Processor** | Принимает сырые данные с камер/микрофонов роботов, выполняет распознавание образов, классификацию, транскрипцию речи. Публикует события о распознанных объектах, командах, аномалиях. Может использовать как локальные модели (для быстрой реакции), так и централизованные (для сложного анализа). |

## 6. AI-обработка: архитектурные решения

### 6.1. Два режима работы AI-модуля

1. **Периферийный (Edge AI)** – модель выполняется непосредственно на роботе (например, на встроенном GPU или NPU). Это даёт минимальную задержку, критичную для навигации и избегания препятствий.
2. **Централизованный (Cloud/Backend AI)** – робот отправляет сжатые кадры или аудиофрагменты на бэкенд, где выполняется сложная обработка. Используется для задач, требующих больших вычислительных ресурсов (например, распознавание лиц, детальный анализ сцен).

Оркестратор может динамически выбирать режим в зависимости от типа задачи, доступной сети и заряда батареи.

### 6.2. Интеграция AI с оркестратором

- AI-модуль подписывается на события `robot.status.changed` и `task.assigned`, чтобы знать, какие роботы активны и какие задачи требуют AI-обработки.
- При распознавании объекта или команды AI публикует событие `ai.object.detected` или `ai.speech.command`, на которое может реагировать Task Dispatcher (например, повысить приоритет задачи или изменить маршрут).
- Все AI-события логируются и отправляются в Analytics для обучения моделей и улучшения алгоритмов.

### 6.3. Выбор технологий (перспективный)

- **Распознавание изображений**: YOLO (или его аналоги) для обнаружения объектов; OpenCV для предобработки.
- **Видеоаналитика**: DeepStream (NVIDIA) или собственная реализация на Go с использованием библиотек компьютерного зрения.
- **Распознавание голоса**: Whisper (OpenAI) или Vosk (для офлайн-режима).
- **Интерпретация**: централизованный сервис может использовать LLM для понимания контекста команд (например, «доставь товар в зону 5»).

## 7. Хранилища

- **PostgreSQL** – транзакционные данные (роботы, задачи, локации, история событий).
- **Redis** – кэш состояний роботов, сессии, распределённые блокировки.
- **ClickHouse** (этап 2) – аналитика, агрегация метрик, хранение AI-событий для обучения.
- **S3-совместимое хранилище** (этап 2) – для хранения видео/аудио/изображений, переданных роботами.

---

## 8. Интеграции

- **С роботами** – через MQTT или WebSockets (в зависимости от типа робота). В первой версии – только эмулятор.
- **С внешними системами** – REST/gRPC API для заказчиков, складов, систем мониторинга.
- **С AI-библиотеками** – через внутренние вызовы (в монолите) или через отдельный gRPC-сервис (при распиле).

## 9. Unity 6 – цифровой двойник

Подробно описано в отдельном документе `emulator.md`. Кратко:
- Unity выступает как «движок физики и визуализации».
- Получает команды от бэкенда через WebSocket, имитирует движение роботов и возвращает телеметрию.
- Позволяет тестировать алгоритмы в реалистичной среде до запуска на реальном железе.
- Также может симулировать работу AI-сенсоров (генерация синтетических данных для обучения).

## 10. Безопасность, наблюдаемость, обработка сбоев

- **Безопасность**: JWT-аутентификация для внешних API, TLS для связи с роботами, шифрование AI-данных.
- **Мониторинг**: Prometheus + Grafana (метрики запросов, времени ответа, состояния роботов, загрузки AI-модулей).
- **Логирование**: структурированные логи (JSON) с уровнями, отправка в централизованную систему.
- **Сбои**:
  - Потеря связи – ретраи, fallback-режимы.
  - Перегрузка – rate limiting, очереди.
  - Сбой робота – перераспределение задач.
  - Сбой AI-обработки – возврат к базовым алгоритмам навигации (без AI).

## 11. Команда, лидерство и процессы

**Роль**: Tech Lead / Team Lead / Software Architect (полный цикл управления продуктом).

**Основные обязанности**:
- **Архитектурное проектирование**: разработка и защита архитектурных решений, документация, управление техническим долгом.
- **Управление командой**: планирование спринтов, декомпозиция задач, приоритизация, контроль выполнения, code review.
- **Найм и развитие команды**: проведение технических собеседований, онбординг новых разработчиков, менторинг, построение карьерных треков.
- **Развитие продукта**: взаимодействие со стейкхолдерами, перевод бизнес-требований в технические задачи, формирование видения продукта на 6–12 месяцев вперёд.
- **Инженерные практики**: внедрение и поддержка CI/CD, автоматического тестирования, мониторинга, стандартов кода.
- **Документирование**: ведение архитектурной документации, ADR, API-контрактов, руководств для команды.

**Команда (план роста)**:
- На старте (MVP): 3–5 разработчиков Go + 1 инженер Unity (в роли Давида на старте, далее — выделенный специалист).
- На этапе масштабирования (6–12 месяцев): расширение до 10+ инженеров, формирование выделенных подкоманд (бэкенд, инфраструктура, симуляция, AI).

**Процессы**:
- **Git Flow** — ветки для фич, релизов, хотфиксов.
- **Code Review** — обязательное ревью каждого PR как минимум одним участником команды.
- **CI/CD** — автоматическая сборка, тестирование, деплой в staging при каждом push, в production — по тегу.
- **Тестирование** — unit (>70% покрытия), интеграционные тесты (testcontainers), нагрузочные (k6/vegeta).
- **Мониторинг** — Prometheus + Grafana, алертинг в Telegram/Slack.
- **Документирование** — всё в репозитории, актуализация после каждого крупного изменения.

# Риски и компромиссы (Trade-offs)

## 1. Риски и их митигация

### 🔴 Технические риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **Потеря связи с роботом** | Высокая | Критическое | Retry с экспоненциальной задержкой → перевод робота в offline → перераспределение задач. Graceful degradation — система продолжает работу с оставшимися роботами. |
| **Перегрузка оркестратора** | Средняя | Высокое | Rate limiting, горизонтальное масштабирование (множество экземпляров Task Dispatcher). Использование очередей (Kafka) для амортизации пиковых нагрузок. |
| **Сложность распила на микросервисы** | Средняя | Высокое | Строгое разделение по DDD с самого начала. Чёткие интерфейсы и событийная коммуникация (InMemoryBus → Kafka). ADR для каждого решения. Регулярные архитектурные ревью. |
| **Unity-симуляция ресурсоёмка** | Средняя | Среднее | Использование DOTS, ECS, Burst Compiler, C# Job System для обработки тысяч объектов. LOD (уровни детализации) и упрощённые модели для массовых объектов. |
| **Задержки в распределённых транзакциях** | Средняя | Высокое | Использование Saga (оркестрация + хореография) для консистентности. Идемпотентность операций. Event sourcing для восстановления состояния. |
| **Недостаточная производительность БД** | Низкая | Среднее | Индексирование, партиционирование, шардирование (Consistent Hashing). Репликация. Кэширование в Redis. ClickHouse для аналитики. |
| **Безопасность** (подмена команд, утечка данных) | Низкая | Критическое | JWT-аутентификация, TLS для связи с роботами, шифрование AI-данных. Аудит логов. Регулярное обновление зависимостей. |
| **Сбой в AI-модуле** | Низкая | Среднее | Периферийный AI продолжает работу. Централизованный AI отключается до восстановления. Оператор получает уведомление и переходит на ручное управление (Human-in-the-loop). |

---

### 🟡 Организационные риски

| Риск | Вероятность | Влияние | Митигация |
|------|-------------|---------|-----------|
| **Команда не успевает по срокам** | Средняя | Высокое | Приоритизация задач, MVP-first подход. Гибкое планирование спринтов. Найм дополнительных инженеров по мере роста. |
| **Недостаток навыков в Unity** | Средняя | Среднее | На старте достаточно одного инженера (есть у меня). Позже — формирование выделенной команды. Документация и обучение. |
| **Недостаток навыков в AI** | Средняя | Среднее | Использование готовых моделей (YOLO, Whisper). Найм ML-инженеров на этапе 2. Аутсорсинг для сложных задач (если нужно). |
| **Изменение требований бизнеса** | Высокая | Среднее | Инкрементальный подход, гибкая архитектура. Регулярные встречи со стейкхолдерами. Feature flags для быстрого включения/отключения функций. |
| **Бюрократия и сопротивление изменениям** | Средняя | Среднее | Документирование решений (ADR), демонстрация быстрых результатов (MVP), вовлечение команды в процесс принятия решений. |

## 2. Ключевые компромиссы (Trade-offs)

Каждое архитектурное решение связано с компромиссами. Мы осознанно выбираем их, исходя из бизнес-целей и текущих ограничений.

### ⚖️ Модульный монолит vs Микросервисы

| Критерий | Модульный монолит | Микросервисы |
|----------|-------------------|--------------|
| **Time-to-Market** | ✅ Быстрый (2–3 месяца) | ❌ Медленный (6+ месяцев) |
| **Сложность разработки** | ✅ Низкая | ❌ Высокая (распределённые транзакции) |
| **Масштабируемость** | ⚠️ Горизонтально только целиком | ✅ Точечно |
| **Надёжность** | ❌ Ошибка в модуле может уронить всё | ✅ Отказ изолирован |
| **Гибкость стека** | ❌ Один язык/БД | ✅ Каждый сервис своё |
| **DevOps нагрузка** | ✅ Минимальная | ❌ Высокая |
| **Сложность распила** | ⚠️ Требует дисциплины | ✅ Готово к распилу |

**Вывод:** мы выбираем модульный монолит как стратегический старт, чтобы быстро выкатить MVP и минимизировать риски, но с чёткими границами для будущего распила.

---

### ⚖️ Событийная архитектура vs Синхронные вызовы

| Критерий | Event-Driven | Синхронные вызовы (REST/gRPC) |
|----------|--------------|-------------------------------|
| **Связанность** | ✅ Слабая | ❌ Жёсткая |
| **Асинхронность** | ✅ Да | ❌ Нет (блокирующие) |
| **Отказоустойчивость** | ✅ Высокая | ❌ Низкая (каскадные отказы) |
| **Сложность отладки** | ❌ Высокая (асинхронные цепочки) | ✅ Низкая (прямые вызовы) |
| **Масштабируемость** | ✅ Легко добавлять подписчиков | ⚠️ Ограничена нагрузкой на API |
| **Идемпотентность** | ❌ Требуется явная | ✅ Проще обеспечить |

**Вывод:** событийная архитектура даёт нам гибкость и отказоустойчивость, критичные для системы с тысячами роботов. Цена — сложность отладки, которую мы компенсируем трейсингом и логированием.

---

### ⚖️ InMemoryBus vs Kafka

| Критерий | InMemoryBus | Kafka |
|----------|-------------|-------|
| **Скорость** | ✅ < 1 мс | ⚠️ 5–10 мс |
| **Сложность** | ✅ Минимальная | ❌ Высокая (настройка, поддержка) |
| **Надёжность** | ❌ Нет сохранности | ✅ Сохраняет события на диске |
| **Масштабируемость** | ❌ Только в рамках одного процесса | ✅ Горизонтально |
| **Потоки** | ❌ Нет (только pub/sub) | ✅ Поддержка потоков (Kafka Streams) |

**Вывод:** InMemoryBus — для монолита, Kafka — при распиле. Модули не знают, какая шина используется, благодаря абстракции `EventBus`.

---

### ⚖️ PostgreSQL vs NoSQL (MongoDB, Cassandra)

| Критерий | PostgreSQL | NoSQL |
|----------|------------|-------|
| **ACID** | ✅ Гарантирует | ❌ Обычно нет (eventual consistency) |
| **Схема** | ❌ Жёсткая (миграции) | ✅ Гибкая (схема-on-read) |
| **Масштабируемость** | ⚠️ Вертикальная + шардирование | ✅ Горизонтальная (из коробки) |
| **Индексы** | ✅ Мощные (B-Tree, GIN, GiST) | ⚠️ Ограничены |
| **Сложность** | ✅ Стандартный инструмент | ❌ Требует обучения |

**Вывод:** выбираем PostgreSQL для транзакционных данных (роботы, задачи), так как нам важны консистентность и сложные запросы. Для аналитики — ClickHouse. Для кэша — Redis.

---

### ⚖️ Edge AI vs Cloud AI

| Критерий | Edge AI (на роботе) | Cloud AI (на бэкенде) |
|----------|---------------------|-----------------------|
| **Задержка** | ✅ < 50 мс | ❌ 100–300 мс |
| **Зависимость от сети** | ❌ Нет | ❌ Есть |
| **Вычислительные ресурсы** | ❌ Ограничены (GPU/NPU) | ✅ Мощные (GPU-кластер) |
| **Обновление моделей** | ❌ Сложно (требует прошивки) | ✅ Просто (централизованно) |
| **Безопасность** | ✅ Данные не передаются | ❌ Данные передаются по сети |

**Вывод:** используем гибридный подход — периферийный AI для быстрых задач (обнаружение препятствий), централизованный AI для сложных (видеоаналитика, обучение). Оркестратор выбирает режим динамически.

---

### ⚖️ Unity 6 vs Собственная симуляция

| Критерий | Unity 6 | Собственная симуляция |
|----------|---------|------------------------|
| **Скорость разработки** | ✅ Высокая (готовый движок) | ❌ Низкая (писать с нуля) |
| **Физика** | ✅ PhysX (промышленный стандарт) | ❌ Своя реализация (риск ошибок) |
| **Визуализация** | ✅ Фотореалистичная | ❌ Ограниченная |
| **Кроссплатформенность** | ✅ Да (Windows, Linux, WebGL) | ❌ Только для целевой платформы |
| **Поддержка сообщества** | ✅ Огромная | ❌ Нет |

**Вывод:** Unity 6 даёт нам готовый движок с физикой и визуализацией, что ускоряет разработку в разы. Единственный риск — производительность, но DOTS + ECS решают эту проблему.

---

### ⚖️ Oracle vs Open Source (PostgreSQL, ClickHouse)

| Критерий | Oracle | Open Source (PostgreSQL, ClickHouse) |
|----------|--------|---------------------------------------|
| **Стоимость** | ❌ Очень высокая | ✅ Бесплатно |
| **Поддержка** | ✅ Коммерческая | ⚠️ Сообщество / платный SLA |
| **Производительность** | ✅ Высокая | ✅ Высокая (для OLTP/OLAP) |
| **Масштабируемость** | ✅ Горизонтальная (RAC) | ⚠️ Требует дополнительных решений |
| **Гибкость** | ❌ Закрытая экосистема | ✅ Открытая, интеграции |

**Вывод:** в условиях импортозамещения и экономии бюджета выбираем Open Source решения (PostgreSQL, ClickHouse, Redis). Они покрывают все потребности RMS и имеют сильное сообщество.

## 3. Бизнес-ценность управления рисками

Управление рисками — это не просто «техническая необходимость», а **инструмент экономии**:

| Риск | Стоимость без митигации | Стоимость с митигацией |
|------|--------------------------|-------------------------|
| **Потеря связи с роботом** | Простой робота (10 000 руб./час) | Retry + перераспределение (500 руб./час) |
| **Перегрузка оркестратора** | Падение системы (1 млн руб./час) | Rate limiting + масштабирование (1000 руб./час) |
| **Безопасность (утечка данных)** | Штрафы, потеря репутации (50 млн руб.) | JWT + TLS + аудит (1 млн руб.) |
| **Unity-симуляция** | Поломка реального робота (300 000 руб.) | Тестирование в симуляции (10 000 руб.) |

**Итог:** инвестиции в архитектуру и управление рисками окупаются многократно.

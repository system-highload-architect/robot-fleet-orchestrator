# Дорожная карта развития RMS

## 1. Общий подход

Дорожная карта построена на принципах **эволюционной архитектуры**:
- **Инкрементальные изменения** — каждый этап добавляет ценность без переписывания системы с нуля.
- **Фитнес-функции** — проверяют, что система сохраняет ключевые свойства (производительность, надёжность, безопасность).
- **Обратная связь** — после каждого этапа собираем метрики и корректируем план.

---

## 2. Этапы и вехи

### 🚀 Этап 0: MVP (1–2 месяца)

**Цель:** Запустить работающий прототип, который может управлять парком эмулируемых роботов.

**Что делаем:**
- **Архитектура:** модульный монолит с in-memory шиной событий.
- **Бэкенд:** оркестратор (Task Dispatcher), Robot Manager, Location Service (базовые версии).
- **Хранилища:** PostgreSQL (транзакции), Redis (кэш состояний).
- **Эмулятор:** консольный эмулятор роботов (имитация движения, статусов).
- **API:** REST-эндпоинты для внешних систем.
- **Тестирование:** юнит-тесты, интеграционные тесты (in-memory БД).
- **Мониторинг:** базовые метрики (Prometheus + Grafana).

**Результат:** Можно дать команду «доставить груз из А в Б», эмулятор покажет движение, статусы обновляются.

**Ключевые вехи:**
- ✅ Спроектирована архитектура модулей.
- ✅ Реализован базовый оркестратор.
- ✅ Запущен эмулятор.
- ✅ Написаны тесты (покрытие > 70%).

**Фитнес-функции:**
- **Триггерная:** тесты проходят при каждом коммите.
- **Непрерывная:** мониторинг времени отклика API (< 100 мс).
- **Временная:** система запускается за < 30 с.

---

### 🚀 Этап 1: Расширение (3–6 месяцев)

**Цель:** Перейти к реальным роботам, внедрить шину событий на базе Kafka и начать симуляцию в Unity.

**Что делаем:**
- **Интеграция с реальными роботами:** через MQTT / WebSockets (в зависимости от типа).
- **Шина событий:** замена in-memory шины на Kafka (подготовка к распилу).
- **Unity-симуляция:** простая 3D-симуляция с кубами-роботами, управление через WebSocket.
- **AI-модуль:** интеграция готовых моделей (YOLO для обнаружения препятствий, Whisper для голоса).
- **Аналитика:** ClickHouse для сбора метрик (агрегаты по роботам, задачам).
- **Тестирование:** интеграционные тесты с Kafka, нагрузочные тесты (1000+ роботов в симуляции).

**Результат:** Система управляет 10–20 реальными роботами на складе, плюс симуляция сотен роботов в Unity. AI помогает обнаруживать препятствия.

**Ключевые вехи:**
- ✅ Интеграция с первыми реальными роботами.
- ✅ Kafka работает (потери < 1%).
- ✅ Unity-симуляция запущена и синхронизируется с бэкендом.
- ✅ AI-модуль распознаёт объекты и публикует события.

**Фитнес-функции:**
- **Триггерная:** нагрузочные тесты при каждом релизе (RPS, latency).
- **Непрерывная:** мониторинг потерь пакетов (< 3%), задержек событий (< 50 мс).
- **Временная:** время восстановления после сбоя < 1 минуты.

---

### 🚀 Этап 2: Масштабирование (6–12 месяцев)

**Цель:** Распил на микросервисы, поддержка 1000+ роботов, полноценный цифровой двойник.

**Что делаем:**
- **Распил:** вынос модулей в микросервисы (Robot Manager → `robot-manager`, Task Dispatcher → `task-dispatcher`, Location Service → `location-service`, Analytics → `analytics`).
- **Оркестрация:** Kubernetes для деплоя и масштабирования сервисов.
- **Unity-симуляция:** полноценный цифровой двойник с физикой (DOTS, ECS, Burst).
- **AI-модуль:** централизованный AI на GPU-кластере (видеоаналитика, обучение моделей).
- **Хранилища:** S3 для видео/аудио, ClickHouse для аналитики, шардирование PostgreSQL.
- **CI/CD:** автоматический деплой, канареечные релизы, откат при сбоях.

**Результат:** Готовая к продакшену система, протестированная на симуляторе, запущенная на реальном парке. AI помогает операторам.

**Ключевые вехи:**
- ✅ Все модули вынесены в микросервисы.
- ✅ Kubernetes-кластер работает (автомасштабирование).
- ✅ Полноценный цифровой двойник интегрирован с бэкендом.
- ✅ AI-модуль работает в централизованном режиме.

**Фитнес-функции:**
- **Триггерная:** нагрузочные тесты на микросервисах (100 000+ RPS).
- **Непрерывная:** мониторинг SLA (99.9% доступности).
- **Временная:** время развёртывания нового сервиса < 5 минут.

---

### 🚀 Этап 3: Интеллектуализация (12+ месяцев)

**Цель:** Внедрение роевых алгоритмов, машинное обучение для оптимизации маршрутов и адаптивного поведения.

**Что делаем:**
- **Роевые алгоритмы:** локальные агенты для автономного перераспределения задач при потере связи с центром.
- **ML-модели:** прогнозирование загрузки, профилактика сбоев, оптимизация маршрутов (обучение на исторических данных).
- **Самообучение AI:** использование собранных данных для улучшения моделей распознавания.
- **Расширенная аналитика:** дашборды для менеджеров (KPI, тренды, прогнозы).

**Результат:** Полностью автономная, самооптимизирующаяся система управления роботами.

**Ключевые вехи:**
- ✅ Роевые алгоритмы реализованы и протестированы.
- ✅ ML-модели встроены в оркестратор.
- ✅ Система самообучения AI работает (цикл обратной связи).

**Фитнес-функции:**
- **Триггерная:** проверка качества ML-моделей (точность > 95%).
- **Непрерывная:** мониторинг оптимизации маршрутов (снижение времени доставки на 10%).
- **Временная:** время переобучения модели < 1 часа.

## 3. Ресурсы и команда

| Этап | Команда | Инфраструктура |
|------|---------|----------------|
| **0** | 3–5 разработчиков Go | 1 сервер (Dev) |
| **1** | 5–7 разработчиков + DevOps | 5–10 серверов (Staging) |
| **2** | 10+ разработчиков + SRE | Kubernetes-кластер (Production) |
| **3** | 15+ разработчиков + ML-инженеры | GPU-кластер + ML-платформа |

## 4. Риски и управление

| Риск | Митигация |
|------|-----------|
| **Задержка распила** | Чёткие границы модулей с самого начала, регулярные архитектурные ревью. |
| **Производительность Unity** | DOTS + ECS + Burst, тестирование на разных конфигурациях. |
| **Сложность ML-моделей** | Поэтапное внедрение, использование готовых моделей (YOLO, Whisper). |
| **Изменение требований бизнеса** | Инкрементальный подход, гибкость архитектуры. |

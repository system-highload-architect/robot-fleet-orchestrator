# 01. Обзор архитектуры (Architecture Overview)

## Контекст и бизнес-цели

**Заказчик**: обезличен.  
**Продукт**: платформа управления парком автономных роботов (Robot Management System, RMS).  
**Типы роботов**: доставщики, дроны, погрузчики.  
**Локации**: склады, городские зоны, зоны дронов.

**Бизнес-цели**:
- Автоматизация логистики и снижение затрат.
- Повышение скорости обработки заказов.
- Отказоустойчивость и масштабируемость (от 10 до 1000+ роботов).

**Нефункциональные требования**:
- Доступность: 99.9% (роботы не должны простаивать).
- Латентность: p95 < 100 мс для команд управления.
- Безопасность: аутентификация роботов, шифрование команд.
- Масштабируемость: поддержка роста парка без переписывания архитектуры.

## Высокоуровневая архитектура

Архитектура строится вокруг **модульного монолита** с чёткими границами (DDD) и эволюционным планом перехода к микросервисам.

### Ключевые компоненты

| Компонент | Ответственность |
|-----------|-----------------|
| **Robot Manager** | Управление состоянием роботов, телеметрия, публикация событий статуса. |
| **Task Dispatcher** | Распределение задач, приоритеты, обработка отказов. |
| **Location Service** | Управление локациями, маршрутами и ограничениями. |
| **Analytics** | Сбор метрик, мониторинг SLA, аналитика. |
| **AI Processor** | Распознавание объектов, голосовых команд, видеоаналитика. |
| **Unity 6 Simulation** | Цифровой двойник для тестирования, обучения и маркетинга. |
| **Event Bus** | Абстрактная шина событий (in-memory → Kafka). |

### Коммуникация
- **Внутри монолита**: модули общаются через in-memory шину событий.
- **При распиле**: замена на Kafka без изменения кода модулей.
- **С роботами**: через MQTT/WebSockets (в первой версии — эмулятор).

### Хранилища
| Хранилище | Назначение |
|-----------|------------|
| PostgreSQL | Транзакционные данные (роботы, задачи, локации). |
| Redis | Кэш состояний, сессии, распределённые блокировки. |
| ClickHouse | Аналитика и метрики (этап 2). |
| S3 (MinIO) | Хранение видео/аудио/изображений (этап 2). |

### Почему модульный монолит?
1. **Быстрый выход на рынок** — MVP за 2 месяца.
2. **Минимизация рисков** — не нужно проектировать распределённые транзакции на старте.
3. **Гибкость** — чёткие границы модулей позволяют выносить их в микросервисы без переписывания.

### Почему Event-Driven?
1. **Слабая связанность** — модули не знают друг о друге.
2. **Асинхронность** — обработка событий без блокировок.
3. **Отказоустойчивость** — при сбое одного модуля остальные продолжают работать.

## Архитектурные принципы
1. **Чистая архитектура (Clean Architecture)** — бизнес-логика не зависит от внешних фреймворков.
2. **Domain-Driven Design (DDD)** — чёткие ограниченные контексты (Bounded Contexts).
3. **Эволюционная архитектура** — система проектируется для изменений.
4. **Human-in-the-loop** — оператор может вмешиваться в критических ситуациях через Unity-интерфейс.

## Связь с фитнес-функциями
- **Триггерные**: интеграционные тесты при каждом коммите.
- **Непрерывные**: мониторинг потерь пакетов и задержек.
- **Временные**: проверка восстановления после сбоя за заданное время.

# Robot Fleet Orchestrator

**Платформа для управления парком автономных роботов (Robot Management System, RMS).**

Проект находится на стадии архитектурного проектирования. Реализуется как модульный монолит на Go с эволюционным планом перехода к микросервисной архитектуре и использованием Unity 6 для создания цифрового двойника.

## Ключевые цели
- **Управление роботами разных типов** (доставщики, дроны, погрузчики) в различных локациях (склад, город, зоны дронов).
- **Централизованная координация** с элементами роевого взаимодействия для повышения автономности.
- **Высокая надёжность** и низкие задержки (система реального времени).
- **Тестирование без риска** через симуляцию физического мира в Unity 6.

## Роль и ответственность
Данный проект разрабатывается под руководством **Tech Lead / Team Lead / Architect**, который отвечает за:
- Архитектурное видение и техническую стратегию.
- Управление командой разработки (до 10+ инженеров).
- Найм, менторинг и развитие инженеров.
- Взаимодействие с бизнесом и перевод требований в технические задачи.
- Качество кода, инженерные практики и наблюдаемость системы.

## Быстрый старт
```bash
# Демонстрационный режим (оркестратор + эмулятор)
make demo

# Или по отдельности:
make run          # только оркестратор (в фоне или терминале)
make dev          # быстрый запуск оркестратора без сборки

# Запустить тесты
make test

# Проверить код линтером
make lint
```

> **Важно:** Текущая версия использует **in-memory хранилища** для упрощения запуска. PostgreSQL и Redis пока не задействованы, но их интеграция уже заложена в `internal/infrastructure/postgres` и `internal/infrastructure/redis` — это подготовка к будущим этапам.

Подробная инструкция по запуску — в [launch.md](launch.md).

## 🧭 Быстрый экскурс по проекту

**Демонстрационный прототип** модульного монолита для управления парком роботов. Основные акценты:

### 1. **Архитектура**
- **Модульный монолит с DDD** — каждый модуль (robot-manager, task-dispatcher, location-service, analytics) изолирован и готов к выносу в отдельный микросервис.
- **Event-driven коммуникация** — модули общаются через абстрактную шину событий (`eventbus`), которая может быть in-memory (сейчас) или Kafka (в будущем).
- **Чистая архитектура** — слои: API → Service → Repository → Domain, зависимости направлены внутрь.

### 2. **Ключевые модули**
- **Robot Manager** — управляет роботами (состояние, статус). Публикует событие `robot.status.changed`.
- **Task Dispatcher** — создаёт задачи, назначает их роботам, отслеживает выполнение. Публикует `task.assigned`, `task.completed`.
- **Location Service** — управляет локациями (склады, зоны). Публикует `location.created`, `location.updated`.
- **Analytics** — подписывается на события роботов и задач, агрегирует статистику, публикует `analytics.stats.updated`.

### 3. **Инфраструктура**
- **In-memory Event Bus** — лёгкая реализация шины для монолита.
- **In-memory Repository** — для демонстрации (легко заменить на PostgreSQL/Redis).
- **HTTP API** — стандартные REST-эндпоинты для внешних систем и эмулятора.
- **Конфигурация** — через `pkg/config` с поддержкой YAML и env-переменных.
- **Логирование** — структурированное через `pkg/logger` на основе `slog`.

### 4. **Эмулятор роботов**
- Отдельный компонент (`emulator/`), который имитирует поведение роботов и генерацию задач.
- Использует HTTP-адаптер для общения с оркестратором.
- Может быть заменён на Unity-симуляцию в будущем.

### 5. **Структура кода**
- `cmd/orchestrator/` — точка входа (сборка всех модулей, запуск HTTP-сервера).
- `internal/modules/` — каждый модуль со своей логикой.
- `internal/shared/` — общие доменные модели, события, ошибки.
- `internal/infrastructure/` — технические слои (eventbus, postgres, redis — последние пока заглушки).
- `pkg/` — переиспользуемые пакеты (logger, config, utils).
- `emulator/` — независимый эмулятор.
